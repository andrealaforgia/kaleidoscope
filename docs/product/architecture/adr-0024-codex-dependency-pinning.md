# ADR-0024 — `codex` dependency pinning, in-tree Levenshtein

- **Status**: Proposed
- **Date**: 2026-05-07
- **Author**: `nw-solution-architect` (Morgan, dispatched by Bea;
  ADR finalised by Bea after Morgan stalled mid-write at the same
  pattern that caused ADR-0017's recovery)
- **Feature**: `codex` v0
- **Supersedes**: none
- **Superseded by**: none
- **Related**: ADR-0022 (Codex public API), ADR-0023 (corpus
  regeneration ritual), ADR-0013 (Spark dependency pinning, family
  pin precedent), ADR-0019 (Sieve dependency pinning, AGPL-symmetric
  runtime dep precedent)

## Context

Codex v0 is a Rust library. DISCUSS Q1 locked the library shape; Q2
locked the corpus as hand-written Rust constants generated from
upstream `opentelemetry-semantic-conventions`; Q8 locked the
fuzzy-match implementation as in-tree Levenshtein with no new
dependency.

The dependency posture has three loadbearing pieces:

1. The OTel semconv source: where the corpus regeneration ritual
   reads from at maintainer-trigger time.
2. The Levenshtein implementation: in-tree per Q8, but where in the
   crate tree.
3. The runtime closure: nothing else, per Q8.

This ADR locks all three.

## Decision

### 1. Source of the corpus: `opentelemetry-semantic-conventions = "=0.27"` as a build-time-only dependency

The semconv crate is consumed by the regeneration ritual in
ADR-0023 to read attribute names. It is **not** in Codex's runtime
dependency closure. The regeneration ritual is the only consumer;
the generated artefact at `crates/codex/src/generated/semconv_0_27.rs`
is what ships.

The exact-minor pin `=0.27` matches Spark's pin in ADR-0013 §1 and
the harness's pin on `opentelemetry-proto =0.27`. The whole
Kaleidoscope workspace converges on the same OTel family minor.

`Cargo.toml`:

```toml
# Codex has no runtime dependency on opentelemetry-semantic-conventions.
# The corpus is generated from it at maintainer-trigger time and
# checked into src/generated/. The generator (an xtask binary, see
# ADR-0023) is the only consumer.

[dev-dependencies]
opentelemetry-semantic-conventions = { version = "=0.27", optional = true }

# (or: live entirely in xtask/Cargo.toml; see ADR-0023 for the
# regenerator's exact location.)
```

The corpus regeneration ritual is documented in ADR-0023; the
in-process generator binary at `xtask/regenerate_codex_corpus`
imports `opentelemetry-semantic-conventions` and emits the Rust
`static` slice. The published `crates/codex` crate does not carry
this dependency in its release closure.

### 2. Levenshtein: in-tree pure function

```rust
// crates/codex/src/fuzzy.rs

pub(crate) fn levenshtein(a: &str, b: &str) -> usize {
    // Two-row dynamic programming matrix. The implementation is
    // ~30 lines of straightforward Rust. The corpus is small
    // (low hundreds of attribute names); allocation cost is
    // bounded.
    //
    // Threshold for "did you mean" suggestions is 2, locked at
    // DISCUSS Q5.
}
```

The function is `pub(crate)` — used internally by `SchemaCatalogue`
to populate `LintViolation::nearest_blessed_match`. It is not part
of Codex's public surface.

The DP matrix is allocated as `Vec<usize>` of length
`b.chars().count() + 1`. Stack-allocated buffers via SmallVec or
similar are rejected: the simpler implementation reads cleanly at
review time, the corpus size makes the allocation cost negligible
(~hundreds of bytes per call).

Unicode handling: the implementation iterates over `char`s, not
bytes. Attribute names in OTel semconv are ASCII in practice but the
type signature stays Unicode-correct.

### 3. Runtime closure: empty (beyond the standard library)

Codex v0 has zero runtime dependencies beyond `std`. The library
is:

- `crates/codex/src/lib.rs` — re-exports + `forbid(unsafe_code)`
- `crates/codex/src/catalogue.rs` — `SchemaCatalogue`,
  `BlessedAttribute`
- `crates/codex/src/lint.rs` — `LintReport`, `LintViolation`,
  `ViolationKind`
- `crates/codex/src/fuzzy.rs` — Levenshtein
- `crates/codex/src/generated/semconv_0_27.rs` — checked-in
  corpus (regenerated by xtask)

`Cargo.toml`:

```toml
[package]
name = "codex"
version = "0.1.0"
edition.workspace = true
license = "AGPL-3.0-or-later"
rust-version.workspace = true
description = "Schema authority for Kaleidoscope: OpenTelemetry semantic conventions plus Kaleidoscope-house attributes."
repository = "https://github.com/andrealaforgia/kaleidoscope"
publish = false

[lib]
path = "src/lib.rs"

[dependencies]
# Empty. Codex's runtime closure is intentionally minimal.

[dev-dependencies]
# Test infrastructure only.

[lints]
workspace = true
```

No `tracing` runtime dep at v0: Codex returns `Result<(), LintReport>`
and lets the consumer (Spark) decide how to surface the report. The
warn-mode tracing emit lives in Spark per ADR-0025. Codex stays
emit-free at v0.

## Alternatives considered

### Option A (rejected): runtime dep on `opentelemetry-semantic-conventions`

Make Codex import the semconv crate at runtime and walk its
constants directly. Pros: no regeneration ritual; corpus is always
fresh against the pin. Cons: ties Codex's runtime closure to the
upstream crate's API (which changes between minor versions, see
Spark's appender-tracing experience at ADR-0017); pulls a build
dependency Spark already carries; couples Codex to the upstream's
release cadence.

The checked-in generated artefact decouples Codex from the
upstream's API stability; the maintainer regenerates when the pin
moves and the diff lands in a PR for review. ADR-0023 locks this
ritual.

### Option B (rejected): `strsim` crate for Levenshtein

The `strsim` crate (MIT) is fine. Adding it costs one entry in the
licence-audit table and one transitive dep in the supply chain.
Codex is `AGPL-3.0-or-later` so the licence audit gets stricter;
the in-tree implementation is ~30 lines that can be read at review
time and never touched again.

Sieve's ADR-0019 had a similar choice and kept `xxhash-rust` as a
runtime dep because the algorithm there is non-trivial (a
collision-resistant 64-bit hash with an audited reference
implementation). Levenshtein is at the other end of the
implementation-effort scale; in-tree is cheaper.

### Option C (rejected): `triple-accel` or other vectorised distance crate

Vectorised Levenshtein is meaningful when the input corpus is
millions of strings and the distance is computed in tight loops.
Codex's corpus is in the hundreds and the function is called once
per `LintViolation`, which itself is rare. Vectorisation pays for
itself nowhere here.

### Option D (rejected): embed the full OTel semconv crate in the runtime closure

Carry `opentelemetry-semantic-conventions` as a runtime dep and
lookup attribute names dynamically. Pros: no regeneration ritual.
Cons: same as Option A, plus runtime cost (linear scans through
upstream's constant tables vs O(1) HashSet lookup against the
generated corpus); plus the upstream crate's API surface enters
Codex's break-when-bumped concerns.

The generated artefact path is the canonical pattern. Spark's
`xxhash-rust =0.8` (Sieve's ADR-0019) is the precedent for "pin the
upstream crate version exactly and treat its consumed surface as a
locked artefact"; Codex extends that to "snapshot the consumed
surface into our own tree and skip the runtime dep entirely".

## Consequences

### Positive

- **Zero runtime deps**: Codex's release closure is trivially small.
  `cargo deny check` (Gate 4) has nothing new to audit.
- **AGPL containment is mechanical**: Codex is AGPL; the only
  cross-crate consumer is Spark (Apache-2.0); Spark consumes Codex
  as a runtime dep, and the AGPL on Codex applies to Spark only as
  a runtime dep, not as a derived work — the same posture as Sieve
  / Aperture symmetric AGPL discussed in ADR-0019.
- **Decoupled release cadence**: when the OTel semconv crate
  releases a new minor, Codex's regenerator is run as a maintainer
  ritual; the diff lands in a PR with full visibility. The
  regeneration is opt-in, not implicit.
- **Levenshtein is review-readable**: ~30 lines of pure Rust;
  future maintainers do not need to read external documentation to
  understand the suggestion logic.

### Negative

- **Maintainer ritual is a process burden**: when the OTel semconv
  pin moves at the workspace level, a maintainer must run the
  regenerator. Forgetting causes Codex's corpus to lag the
  workspace pin, leading to false-positive violations on
  newly-blessed attributes. ADR-0023 documents the ritual.
- **Levenshtein implementation drift**: if the in-tree function has
  a bug, the suggestion logic is wrong and operators may follow a
  bad "did you mean" hint. Mutation testing on the function
  (Gate 5, 100% kill rate per ADR-0005) catches obvious mutations
  but not algorithmic mistakes; the slice 05 fixture set of common
  typos is the behavioural defence.

### Trade-off summary

The dependency posture is "pay once at maintainer-ritual time;
freeze for the rest of the v0 lifetime". This trades a small
ongoing process cost (regeneration when the pin moves) for a
materially smaller runtime closure and a much simpler licence
audit.

## Verification

- `cargo deny check` (Gate 4): empty change to the licence-audit
  table; no new dep entries. Verified at the first CI run on
  Codex-touching commits.
- `cargo public-api -p codex` (Gate 2): Codex's public surface
  remains the five items locked in ADR-0022.
- Slice 02 integration test (`slice_02_otel_semconv_corpus`)
  asserts the full upstream OTel semconv 0.27 attribute set is
  blessed in the catalogue, catching corpus drift if the
  regenerator is forgotten.
- Slice 05 integration test (`slice_05_fuzzy_suggestions`) asserts
  known-typo fixtures resolve to the expected suggestions, catching
  Levenshtein bugs.
