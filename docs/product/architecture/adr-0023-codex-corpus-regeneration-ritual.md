# ADR-0023 — `codex` corpus regeneration ritual + generated-file shape

- **Status**: Accepted
- **Date**: 2026-05-07
- **Author**: `@nw-solution-architect` (Morgan)
- **Feature**: `codex` v0
- **Supersedes**: none
- **Superseded by**: none

## Context

DISCUSS Q2 locks "hand-written Rust constants" as the corpus shape for
v0. DISCUSS Q7 locks the *artefact* shape: a checked-in generated
Rust file at `crates/codex/src/generated/{...}.rs`, regenerated by a
maintainer ritual when the upstream
`opentelemetry-semantic-conventions` pin moves. DISCUSS does **not**
lock the regenerator's exact mechanism. DESIGN flag D4 asks Morgan to
choose between a small Rust binary (`xtask`-style), a shell+`jq`
script, and `cargo-script`.

The problem the regenerator solves: turn the upstream semconv crate's
exported constants into Codex's `&'static [BlessedAttribute]` table.
The output is checked into git, so the diff is visible in PR review
when the pin moves (Q7's load-bearing rationale: nothing changes
silently in CI, every corpus delta is a code review event).

The upstream `opentelemetry-semantic-conventions =0.27` crate exposes
its attributes as Rust constants in module-organised form (e.g.
`opentelemetry_semantic_conventions::resource::SERVICE_NAME` is the
literal `"service.name"`). The natural way to enumerate them is to
build a small Rust program that depends on the crate and walks the
relevant modules.

## Decision

### 1. Mechanism — `xtask` Rust binary (D4 resolved)

The regenerator is a Rust binary that reads the upstream
`opentelemetry-semantic-conventions =0.27` crate (via a Cargo
dependency) and emits the generated file. The binary lives at:

```
crates/codex/xtask/                  # or workspace-root xtask/, crafter's choice
├── Cargo.toml                       # declares opentelemetry-semantic-conventions =0.27 dep
└── src/
    └── main.rs                      # writes crates/codex/src/generated/semconv_0_27.rs
```

The exact location (per-crate `xtask/` subdirectory, workspace-level
`xtask/` crate, or a `[[bin]]` section on `crates/codex/Cargo.toml`
gated behind a `regenerate` feature) is the crafter's call. Either
shape satisfies the "checked-in artefact + visible regeneration
ritual" contract from Q7.

The maintainer invokes the regenerator via:

```sh
cargo run --package codex-xtask -- regenerate
# or, if a feature-gate shape is chosen:
cargo run --package codex --features regenerate --bin codex-regen
```

The regenerator:

1. Reads the v0.27 resource-class constants from
   `opentelemetry_semantic_conventions::resource::*`.
2. Filters to resource-attribute-class entries (excluding trace,
   metric, and log scoped attributes — Q-out-of-scope per
   slice-02-otel-semconv-corpus.md).
3. Emits `crates/codex/src/generated/semconv_0_27.rs` containing a
   `pub(crate) const SEMCONV_0_27: &'static [BlessedAttribute] = &[
   BlessedAttribute::Exact("service.name"), ... ];` slice.
4. Runs `rustfmt` on the emitted file so the diff is review-friendly.
5. Prints a summary line ("regenerated 87 entries from
   opentelemetry-semantic-conventions =0.27.0") so the maintainer can
   sanity-check the output before committing.

The binary itself is not part of the published `codex` crate's surface;
it is a contributor-only tool.

### 2. Generated-file shape

```rust
// crates/codex/src/generated/semconv_0_27.rs
//
// AUTO-GENERATED by codex-xtask from
// opentelemetry-semantic-conventions =0.27.0 — DO NOT EDIT BY HAND.
// To regenerate, run:
//
//     cargo run --package codex-xtask -- regenerate
//
// Diff this file in PR review when the upstream pin moves; the diff
// is the audit trail of what attributes the catalogue gained or lost.

use crate::catalogue::BlessedAttribute;

pub(crate) const SEMCONV_0_27: &[BlessedAttribute] = &[
    BlessedAttribute::Exact("deployment.environment"),
    BlessedAttribute::Exact("host.arch"),
    BlessedAttribute::Exact("host.name"),
    BlessedAttribute::Exact("os.type"),
    BlessedAttribute::Exact("process.pid"),
    BlessedAttribute::Exact("process.runtime.name"),
    BlessedAttribute::Exact("service.instance.id"),
    BlessedAttribute::Exact("service.name"),
    BlessedAttribute::Exact("service.namespace"),
    BlessedAttribute::Exact("service.version"),
    BlessedAttribute::Exact("telemetry.sdk.language"),
    // ... (full set ~80–120 entries, exact count documented in the
    //      slice-02 acceptance test)
];
```

Sorted alphabetically by attribute name so the diff on regeneration is
minimal when one attribute is added or removed. Entries are all
`BlessedAttribute::Exact(...)` because the upstream semconv crate
exposes only literal names (no prefix patterns); the prefix variant
`BlessedAttribute::Prefix("feature_flag.")` is added by the catalogue
constructor in `catalogue.rs` alongside the three house attributes.

### 3. Catalogue construction

```rust
// crates/codex/src/catalogue.rs (sketch)

use crate::generated::semconv_0_27::SEMCONV_0_27;

const HOUSE_ATTRIBUTES: &[BlessedAttribute] = &[
    BlessedAttribute::Exact("tenant.id"),
    BlessedAttribute::Prefix("feature_flag."),
    BlessedAttribute::Exact("experiment.id"),
];

impl SchemaCatalogue {
    pub fn new() -> Self {
        // The crafter chooses whether to pre-build a sorted index
        // (binary search) or scan linearly. Both are fine at ~120
        // entries.
        SchemaCatalogue { /* ... */ }
    }

    pub(crate) fn entries(&self) -> impl Iterator<Item = &BlessedAttribute> {
        SEMCONV_0_27.iter().chain(HOUSE_ATTRIBUTES.iter())
    }
}
```

The catalogue's effective set is the concatenation of `SEMCONV_0_27`
and `HOUSE_ATTRIBUTES`. The two slices are kept separate at v0 so the
regenerator never touches the house attributes (they are hand-written
in `catalogue.rs`); a regeneration that misbehaves and clobbers
`semconv_0_27.rs` cannot accidentally drop the house attributes.

### 4. Pin-bump procedure

When the workspace bumps `opentelemetry-semantic-conventions` (i.e.
when Spark or Aperture's OTel family pin moves):

1. Bump the pin in `crates/codex/Cargo.toml` (and the xtask's
   `Cargo.toml`) in the same PR that bumps Spark / Aperture.
2. Run `cargo run --package codex-xtask -- regenerate`.
3. Review the diff in `crates/codex/src/generated/semconv_0_27.rs`
   (rename the file if the pin family changes, e.g.
   `semconv_0_28.rs`).
4. Run `cargo test -p codex --test slice_02_otel_semconv_corpus` to
   confirm the regenerated corpus still passes the slice-02
   acceptance test (the test asserts a representative spread of
   semconv attributes validate clean).
5. Commit the regenerated file alongside the pin bump.

The procedure is documented in `crates/codex/README.md` (or rustdoc on
`SchemaCatalogue::new`) so a new contributor can run it without
reading the ADR.

## Considered Alternatives

### Alternative 1 — `build.rs` regenerating from the upstream crate at compile time

Pros: source of truth always matches the pinned crate version; no
maintainer ritual to forget; no risk of the generated file lagging the
pin.

Cons (this is the DISCUSS Q7 lock's rejected alternative; restated
here for completeness): build-speed regression for every developer's
incremental builds (the generator runs on every `cargo build` in a
clean target/); PR-diff visibility loss (a pin bump that adds or
removes attributes silently changes the corpus, with the only audit
trail being the `Cargo.lock` diff). For a feature whose whole purpose
is *strictness about attribute names*, the silent-corpus risk is the
exact failure mode the lint exists to prevent.

**Rejected**.

### Alternative 2 — Shell + `jq` script reading the upstream crate's source

Pros: no Rust compile dependency; runnable on any developer machine
with a shell.

Cons: the upstream representation is Rust source, not JSON. The
script would have to either (a) parse Rust source with regex, brittle
when the upstream layout changes between minor versions, or (b) build
the upstream crate first and read `cargo metadata` JSON, which still
needs a Rust toolchain and now also needs `jq`. The Rust binary
alternative pays the same Rust-toolchain cost without the parsing
brittleness, with no `jq` dependency, and with full type safety on the
emitted file.

**Rejected**.

### Alternative 3 — `cargo-script` one-off

Pros: a single `.rs` file that runs as a script via
`cargo-script`-style tooling.

Cons: adds a `cargo-script` (or `rust-script`) installation step to
the contributor onboarding ritual for a single-purpose tool used once
per upstream pin bump. The xtask binary uses only the standard Cargo
tooling already required to build the workspace; contributor surface
stays unchanged.

**Rejected**.

### Alternative 4 — Hand-typed list

Pros: smallest possible mechanism; no tooling at all.

Cons: ~80–120 attribute names; transcription errors are exactly the
failure mode the lint exists to prevent. The whole point of the
regenerator is that the corpus is mechanically derived from a typed
source.

**Rejected**.

## Consequences

**Positive**:
- The corpus is always traceable to the pinned upstream version via
  the file header comment and the regenerator binary's source.
- PR diff on the generated file is the audit trail of what the
  catalogue gained or lost on each pin bump.
- Developer build speed unaffected by the generator; the generator
  runs only on the maintainer's pin-bump PR.
- The generated file is ordinary Rust source, so it compiles with the
  workspace, type-checks, and contributes to mutation testing the
  same way any other module does.
- The `xtask` mechanism mirrors a well-known Rust convention
  (`cargo-xtask` is widely used); contributors recognise the shape.

**Negative**:
- A maintainer must remember to run the regenerator when bumping the
  pin. Mitigation: the slice-02 acceptance test fails if the
  regenerated corpus is missing attributes the test exercises; CI
  catches the regression on the same PR that bumps the pin.
- The xtask binary is a small additional contributor-only artefact
  to maintain. Mitigation: the binary's logic is ~50 lines (read
  module, filter, emit). The maintainer cost is low and is paid once
  per upstream pin bump (yearly cadence at most for OTel semconv).

**Trade-offs**:
- Audit-trail visibility (the explicit goal) vs build-time
  automation (the rejected alternative). DISCUSS Q7 already locked
  the trade-off in favour of audit-trail visibility; this ADR
  inherits that posture.

## Quality attribute alignment

- **Maintainability**: regenerator is one short binary; the generated
  file is sorted alphabetically for minimal diff churn.
- **Reliability**: the slice-02 acceptance test covers the corpus's
  completeness against a fixture set of common semconv attributes.
- **Compatibility**: pin-bump procedure is one PR; corpus rename
  (e.g. `semconv_0_27.rs` → `semconv_0_28.rs`) is a search-and-replace
  on the import line in `catalogue.rs`.
- **Auditability** (cross-cutting): the file header comment on the
  generated file names the upstream version; PR diff visualises the
  delta; the slice-02 test enforces coverage.
