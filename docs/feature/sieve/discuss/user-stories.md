# Sieve v0 — user stories

Six LeanUX user stories with mandatory Elevator Pitches per the
nWave DISCUSS template. Personas are drawn from a fictional but
internally consistent observability platform team at `acme-observability`.

The principal user is **Riley, a site reliability engineer** running
Kaleidoscope in production. Riley's job is to keep storage costs in
check while preserving the trace data that matters when an incident
fires. Secondary user is **Sasha, a platform engineer** wiring
Kaleidoscope into an existing OTel pipeline; Sasha needs Sieve to
slot in cleanly without disrupting Aperture's contract.

System constraints (apply to every story):

1. Library, not service. Sieve v0 is a Rust crate Aperture's pipeline
   consumes; not a separate process binary (per Q1).
2. AGPL-3.0-or-later. Same rationale as Aperture per `LICENSING.md`.
3. Trace-level sampling decisions only at v0; per-span decisions
   rejected (per Q2).
4. Error span definition is `status.code == ERROR` (per Q3); a trace
   is error-bearing if it contains any error span; error-bearing
   traces are retained at 100%.
5. Single global sample rate via `SIEVE_NON_ERROR_TRACE_RATE` env var
   in `[0.0, 1.0]`, default `0.1` (per Q5).
6. Logs and metrics pass through unfiltered at v0 (per Q6).
7. PII-scrubbing is v1+; no scrubbing logic in v0 (per Q4).
8. `xxh3_64` is the hash function for `trace_id`-keyed determinism
   (per Q7).
9. Per-trace sampling-decision events at DEBUG; aggregate summary at
   INFO on periodic tick (per Q8).
10. Public surface is the `Sampler` trait plus its concrete
    `HeadSampler` implementation (DESIGN may adjust the exact shape).

## US-SI-01 — Walking skeleton: error trace kept, non-error trace dropped

### Elevator Pitch

- **Before**: Sieve does not exist as a code artefact; Aperture's
  pipeline has no hook for sampling; SRE Riley cannot demonstrate a
  single sampling decision end-to-end on a fixture trace.
- **After**: run `cargo test -p sieve --test slice_01_walking_skeleton` →
  sees a GREEN test result that proves one error-bearing trace is
  kept (Decision::Keep) and one non-error trace at rate 0.0 is
  dropped (Decision::Drop). The contract is observable as a typed
  Rust enum value asserted by the test.
- **Decision enabled**: Riley decides Sieve's basic shape is correct
  and the team can build subsequent slices on top. Sasha decides
  Aperture's pipeline integration point is sound.

### Problem

A sampling component that nobody has run end-to-end is conjecture.
The walking skeleton is the smallest unit of evidence that the
contract has shape, the trait compiles, and the keep / drop decision
is observable at the call site.

### Who

- Riley, SRE: needs the proof of concept before approving further
  slices.
- Sasha, platform engineer: needs to see the integration shape
  before committing to wiring Sieve into the live Aperture pipeline.

### Solution

Define a `Sampler` trait with one method that takes a trace's spans
and returns a `Decision::Keep` or `Decision::Drop`. Define a
`HeadSampler` concrete implementation that takes a non-error rate at
construction. Run an integration test that builds two fixture traces
(one error-bearing, one not), runs them through `HeadSampler::new(0.0)`,
and asserts the decisions.

### Domain examples

1. **Acme prod incident playback**: Riley plays back a captured trace
   stream from last week's checkout-service outage. The trace
   carries one span with `status.code == ERROR` and an error message
   "downstream timeout". `HeadSampler::new(0.0)` returns
   `Decision::Keep` for this trace because the error-bias kicks in.
2. **Synthetic load test trace**: a synthetic span with `status.code
   == OK`, generated by Sasha's load test on `payments-api`, runs
   through `HeadSampler::new(0.0)` and returns `Decision::Drop`. The
   trace is not preserved.

### UAT scenarios (BDD)

#### Scenario: An error-bearing trace is always kept

```
Given a HeadSampler constructed with non-error rate 0.0
And a fixture trace whose root span carries status.code = ERROR
When the sampler is asked for a decision on that trace
Then the decision is Decision::Keep
```

#### Scenario: A non-error trace is dropped at rate 0.0

```
Given a HeadSampler constructed with non-error rate 0.0
And a fixture trace whose spans all carry status.code = OK
When the sampler is asked for a decision on that trace
Then the decision is Decision::Drop
```

### Acceptance Criteria

- [ ] `crates/sieve/Cargo.toml` declares the crate as workspace member with `license = "AGPL-3.0-or-later"`.
- [ ] `Sampler` trait is exposed on the public surface with one method returning `Decision`.
- [ ] `HeadSampler` is a concrete `pub struct` with a constructor taking a non-error rate as `f64`.
- [ ] `Decision::Keep` and `Decision::Drop` are the two variants of a public `enum`.
- [ ] One integration test asserts an error-bearing trace yields `Keep`.
- [ ] One integration test asserts a non-error trace at rate 0.0 yields `Drop`.

### Outcome KPIs

- 100% of error-bearing fixture traces yield `Decision::Keep` (CI invariant).
- 100% of non-error fixture traces at rate 0.0 yield `Decision::Drop` (CI invariant).
- Walking-skeleton test runs in under 1 second on the runner (no network, no cross-process IO).

### Technical Notes

DESIGN-wave decisions: exact `Sampler::sample` signature shape (slice
of spans vs iterator vs Aperture's batch type); whether `Decision`
is a sealed enum or carries metadata (e.g. reason code).

## US-SI-02 — Error-bias retention: errors always survive sampling

### Elevator Pitch

- **Before**: with only the walking skeleton, Sieve only proves rate
  0.0 drops non-error traces; it does not yet prove the error-bias
  rule holds across non-zero rates.
- **After**: run `cargo test -p sieve --test slice_02_error_bias` →
  sees an error-bearing trace return `Decision::Keep` regardless of
  the configured non-error rate (tested at 0.0, 0.1, 0.5, 1.0).
- **Decision enabled**: Riley decides error retention is the
  load-bearing guarantee Sieve is built around, and Sieve can be
  trusted to never sample errors away.

### Problem

The whole point of error-bias is that errors are the trace data
operators most want to keep. If Sieve loses a single error trace at
any non-error rate, the operator's confidence in sampling collapses.
This story makes that guarantee a CI invariant.

### Who

Riley, SRE: needs the guarantee that an error trace is never dropped
by sampling, regardless of rate.

### Solution

Extend `HeadSampler` to detect error-bearing traces: a trace is
error-bearing if any span in it has `status.code == ERROR`. When
`is_error_bearing(trace) == true`, return `Decision::Keep`
unconditionally; otherwise apply the rate-based decision from
slice 03.

### Domain examples

1. Database timeout span at `acme-prod`: a single span with
   `status.code == ERROR` and `db.system = postgresql`. Trace is
   kept at every tested rate.
2. Multi-span trace with one error: a checkout-service trace with
   12 spans, one of which has `status.code == ERROR` (an upstream
   call to inventory-service timed out). Whole trace is kept.
3. All-OK trace: a successful `payments-api` call with 5 OK spans.
   Subject to the rate-based decision (slice 03's territory; not
   asserted here).

### UAT scenarios (BDD)

#### Scenario: An error trace is kept at every non-zero rate

```
Given a HeadSampler constructed with non-error rate <r>, where <r> ∈ {0.0, 0.1, 0.5, 1.0}
And a fixture trace containing at least one span with status.code = ERROR
When the sampler is asked for a decision
Then the decision is Decision::Keep
```

#### Scenario: A multi-span trace with one error span is kept

```
Given a HeadSampler with non-error rate 0.0
And a fixture trace with 12 spans, one of which has status.code = ERROR
When the sampler is asked for a decision
Then the decision is Decision::Keep
```

#### Scenario: A trace with no error spans is not retained by the error-bias rule

```
Given a HeadSampler with non-error rate 0.0
And a fixture trace with 5 spans, all of which have status.code = OK
When the sampler is asked for a decision
Then the decision is NOT forced to Keep by the error-bias rule
And the rate-based decision applies (and at rate 0.0 is Drop)
```

### Acceptance Criteria

- [ ] An internal `is_error_bearing(trace)` function exists and returns true iff any span has `status.code == ERROR`.
- [ ] When `is_error_bearing(trace) == true`, the sampler returns `Decision::Keep` regardless of rate.
- [ ] At least one parameterised test runs the assertion across rates 0.0, 0.1, 0.5, 1.0 and confirms error retention at all four.

### Outcome KPIs

- 100% of error-bearing fixture traces yield `Decision::Keep` across all four sampled rates (CI invariant).
- The `is_error_bearing` check has 100% mutation kill rate.

### Technical Notes

DESIGN: confirm that `status.code` is reachable from the span shape
Sieve consumes (likely `opentelemetry_proto::tonic::trace::v1::Span`).

## US-SI-03 — Non-error rate honoured statistically

### Elevator Pitch

- **Before**: Sieve can keep error traces; whether non-error traces
  are sampled at the configured rate is unproven.
- **After**: run `cargo test -p sieve --test slice_03_non_error_rate`
  → sees a streamed fixture of 10000 non-error traces with a
  configured rate of 0.5 produce ~5000 kept (within ±3%) and the rest
  dropped. Tests at 0.0, 0.5, 1.0 all pass.
- **Decision enabled**: Riley sees the configured rate is respected
  in aggregate, and trusts Sieve's volume-control story.

### Problem

The configured rate is the operator's volume-control knob. If the
rate is not statistically honoured, Sieve's value proposition fails.
This story makes the statistical guarantee a CI invariant.

### Who

Riley, SRE: trusts the rate to deliver predictable volume reduction.

### Solution

`HeadSampler::sample` for a non-error trace hashes the `trace_id`
with `xxh3_64`, maps the result into `[0.0, 1.0]`, and compares
against the rate. If the mapped value is below the rate, keep;
otherwise drop. A streamed fixture of 10000 traces with random
`trace_id`s is run through the sampler at three rates; the kept
count is asserted within ±3% of the expected (300 / 5000 / 10000
respectively for rates 0.0 / 0.5 / 1.0).

The exact bands and sample size: 10000 traces, ±3% (300 traces) for
rate 0.5; ±2 traces for rate 0.0 and 1.0 (rate 0.0 should keep zero,
rate 1.0 should keep all, but stochastic edge cases at the boundary
are tolerated within ±2). Andrea reviewed and locked these bounds at
DISCUSS time.

### Domain examples

1. **Steady-state production**: 10000 trace IDs sampled at rate 0.1
   → about 1000 kept. The team's monthly storage spend drops by 90%
   on non-error trace volume.
2. **Disabled sampling**: rate 1.0 → all 10000 traces kept.
   Operators use this during incidents to drop the non-error filter.
3. **Total filter**: rate 0.0 → 0 non-error traces kept. Errors only.

### UAT scenarios (BDD)

#### Scenario: At rate 0.0 essentially no non-error traces are kept

```
Given a HeadSampler at rate 0.0
And 10000 distinct fixture trace IDs, all non-error
When each is asked for a decision
Then at most 2 are kept
```

#### Scenario: At rate 1.0 essentially all non-error traces are kept

```
Given a HeadSampler at rate 1.0
And 10000 distinct fixture trace IDs, all non-error
When each is asked for a decision
Then at least 9998 are kept
```

#### Scenario: At rate 0.5 about half the non-error traces are kept

```
Given a HeadSampler at rate 0.5
And 10000 distinct fixture trace IDs, all non-error
When each is asked for a decision
Then between 4700 and 5300 are kept
```

### Acceptance Criteria

- [ ] `HeadSampler::sample` hashes `trace_id` with `xxh3_64` and maps the result into `[0.0, 1.0]`.
- [ ] The kept-count assertion at rate 0.5 lands in `[4700, 5300]` for a 10000-trace fixture (±3% band).
- [ ] The kept-count assertion at rate 0.0 lands in `[0, 2]`.
- [ ] The kept-count assertion at rate 1.0 lands in `[9998, 10000]`.

### Outcome KPIs

- The configured rate is statistically honoured to within ±3% on a 10000-trace fixture (CI invariant; assertion deterministic given the fixture's `trace_id`s).

### Technical Notes

The 10000-trace fixture uses a deterministic seed so the assertion
is non-flaky despite being statistical. DESIGN may move the seed and
sample size into the test infrastructure.

## US-SI-04 — Trace coherence: same `trace_id` always yields the same decision

### Elevator Pitch

- **Before**: a trace's spans may arrive at Sieve in multiple
  batches; without a deterministic rule, batch 1 might be kept and
  batch 2 dropped, producing a broken trace downstream.
- **After**: run `cargo test -p sieve --test slice_04_trace_id_determinism`
  → sees that the same `trace_id` queried multiple times under the
  same `HeadSampler` always yields the same decision; spans of one
  trace are never partitioned across kept and dropped buckets.
- **Decision enabled**: Riley sees that trace coherence is preserved
  under realistic batching; downstream UIs always see whole traces
  or no trace at all.

### Problem

Aperture batches OTLP exports; spans of a single trace can land in
two consecutive batches. If Sieve's decision is non-deterministic
across calls (e.g. randomised), batch 1 might be kept and batch 2
dropped, leaving the downstream UI with a half-rendered waterfall.
The `xxh3_64`-keyed decision is deterministic per `trace_id` and
solves this.

### Who

Riley, SRE: depends on coherent traces in Prism / Tempo / Jaeger
when chasing a latency spike.

### Solution

Already implied by slice 03's hash-based decision. This story makes
the determinism explicit as a CI invariant rather than an implicit
property: query the same `trace_id` 100 times and confirm the
decision never changes.

### Domain examples

1. **Cross-batch trace**: a 20-span trace whose first 12 spans land
   in batch A and last 8 in batch B. Both batches see the same
   `trace_id`; the decision is the same; the downstream UI sees the
   whole trace or nothing.
2. **Re-sent retry**: an OTel SDK retries a failed export; Sieve
   sees the same `trace_id` twice and returns the same decision both
   times.

### UAT scenarios (BDD)

#### Scenario: Same trace_id queried twice yields identical decisions

```
Given a HeadSampler at rate 0.5
And a non-error fixture trace with a specific trace_id
When the sampler is asked for a decision twice
Then both decisions are equal
```

#### Scenario: Same trace_id across 100 queries never flips decision

```
Given a HeadSampler at rate 0.5
And a non-error fixture trace with a specific trace_id
When the sampler is asked for a decision 100 times
Then the decision count for one outcome is exactly 100 and for the other is exactly 0
```

### Acceptance Criteria

- [ ] `HeadSampler::sample` is referentially transparent for a given `trace_id` and `rate`: the same input pair always produces the same output `Decision`.
- [ ] One CI test asserts 100 calls with the same `trace_id` produce identical decisions.

### Outcome KPIs

- 100% determinism: the variance of decision outcomes across 100 calls with the same `trace_id` is zero (CI invariant).

### Technical Notes

The hash function (`xxh3_64`, locked in Q7) is deterministic by
construction; this story is a defence-in-depth assertion that the
property holds at the call site.

## US-SI-05 — Logs and metrics pass through unfiltered

### Elevator Pitch

- **Before**: Sieve only handles traces; logs and metrics either
  have no path through Sieve or risk being dropped by an
  over-aggressive sampler.
- **After**: run `cargo test -p sieve --test slice_05_logs_metrics_passthrough`
  → sees that a fixture log record and a fixture metric data point
  pass through Sieve's pipeline unchanged. The downstream stage
  receives every log and metric record Sieve received.
- **Decision enabled**: Riley deploys Sieve in production knowing
  that traces are sampled but logs and metrics keep their full
  fidelity. Operations dashboards and alerts that depend on
  unsampled metrics keep working.

### Problem

Sampling is for traces. Sieve must not accidentally filter logs or
metrics; v0's volume-control story is trace-only. The pipeline must
have a clear `process_signal(signal)` entry that routes traces to
the sampler and logs/metrics straight through.

### Who

Riley, SRE: depends on log fidelity for incident triage and metric
fidelity for dashboards/SLOs.
Sasha, platform engineer: needs the pipeline contract to be
explicit so the Aperture wiring is clean.

### Solution

Sieve's pipeline accepts a unified `Signal` enum (or three separate
methods, DESIGN's call) covering traces, logs, metrics. The trace
path runs through the `Sampler`; the log and metric paths return
the input unchanged.

### Domain examples

1. **Production log line**: an INFO-level log "checkout completed
   for user U-1234" enters Sieve, exits Sieve unchanged.
2. **Production metric**: a counter `orders_completed_total` with
   value 42 enters Sieve, exits Sieve unchanged.
3. **Mixed batch**: a batch carrying 5 logs, 3 metrics, and 1
   non-error trace at rate 0.0. The 5 logs pass through; the 3
   metrics pass through; the 1 trace is dropped.

### UAT scenarios (BDD)

#### Scenario: A log record passes through Sieve unchanged

```
Given Sieve's pipeline configured with HeadSampler at any rate
And a fixture log record
When the log enters the pipeline
Then the same log exits the pipeline byte-for-byte
```

#### Scenario: A metric data point passes through Sieve unchanged

```
Given Sieve's pipeline configured with HeadSampler at any rate
And a fixture metric data point
When the metric enters the pipeline
Then the same metric exits the pipeline byte-for-byte
```

#### Scenario: Logs are not affected by the trace-rate setting

```
Given Sieve's pipeline configured with HeadSampler at rate 0.0
And 100 fixture log records
When each enters the pipeline
Then 100 logs exit the pipeline
```

### Acceptance Criteria

- [ ] Sieve's pipeline exposes an entry point that accepts traces, logs, or metrics.
- [ ] Logs and metrics are returned to the caller unchanged regardless of the trace rate.
- [ ] At rate 0.0, the log count out equals the log count in (no filtering).
- [ ] At rate 0.0, the metric count out equals the metric count in (no filtering).

### Outcome KPIs

- 100% logs passthrough at every rate (CI invariant).
- 100% metrics passthrough at every rate (CI invariant).

### Technical Notes

DESIGN: pipeline shape — unified `Signal` enum vs three separate
methods. Aperture's `OtlpSink` carries variants per signal, so a
unified enum likely matches; confirm.

## US-SI-06 — Sampling decision observability

### Elevator Pitch

- **Before**: Sieve's decisions are invisible to the operator; if
  the rate is misconfigured (e.g. 0.0 by accident), Riley has no
  way to see it from logs alone.
- **After**: run `cargo test -p sieve --test slice_06_observability`
  → sees DEBUG-level tracing events for each sampling decision
  ("sieve: trace kept (error-bearing)", "sieve: trace dropped
  (rate=0.10)") and a periodic INFO summary
  ("sieve: kept 412 traces (47 error-bearing, 365 sampled at 0.10
  rate), dropped 3614 traces over the last 60s").
- **Decision enabled**: Riley correlates the volume drop with the
  configured rate and notices the misconfiguration before it costs
  a debugging session.

### Problem

Sampling that operators cannot see is sampling they cannot trust.
The observable contract — DEBUG per-decision, INFO summary — gives
operators the visibility they need without flooding their log
aggregator (per Q8).

### Who

Riley, SRE: opts into DEBUG when investigating; reads INFO summary
in normal operation.

### Solution

Each call into `HeadSampler::sample` writes a `tracing` event with
`target = "sieve"` at DEBUG level, whose message describes the
outcome (`kept (error-bearing)` / `kept (sampled, rate=0.10)` /
`dropped (rate=0.10)`). A summary aggregator counts kept/dropped
per minute and emits a single INFO event with the aggregate.

### Domain examples

1. **Investigation mode**: Riley sets `RUST_LOG=sieve=debug` and
   sees every decision streamed during a 10-minute window.
2. **Default mode**: Riley reads the INFO summary every minute and
   compares to the configured rate; misconfiguration is visible in
   the first summary.

### UAT scenarios (BDD)

#### Scenario: A kept error trace emits a DEBUG event

```
Given Sieve's pipeline with the tracing subscriber capturing events for target="sieve"
When the sampler returns Decision::Keep for an error-bearing trace
Then exactly one DEBUG event is emitted with target="sieve" and a message containing "kept (error-bearing)"
```

#### Scenario: A dropped non-error trace emits a DEBUG event

```
Given Sieve's pipeline at rate 0.0
When the sampler returns Decision::Drop for a non-error trace
Then exactly one DEBUG event is emitted with target="sieve" and a message containing "dropped"
```

#### Scenario: A periodic summary aggregates the decisions over the window

```
Given Sieve's pipeline running for at least 60 seconds with mixed traces
When the summary tick fires
Then exactly one INFO event is emitted with target="sieve" containing "kept" and "dropped" counts and the configured rate
```

### Acceptance Criteria

- [ ] Per-trace DEBUG events with `target="sieve"` cover Keep (error-bearing), Keep (sampled), and Drop outcomes.
- [ ] Periodic INFO summary event with `target="sieve"` carries `kept`, `dropped`, and `rate` fields.
- [ ] Per-trace events do NOT appear at INFO or higher (the summary is the operator's default visibility).

### Outcome KPIs

- 100% of sampling decisions emit a DEBUG event (CI invariant).
- 100% of summary windows emit an INFO event (CI invariant).
- Operator can determine the configured rate from a single summary event without reading config.

### Technical Notes

DESIGN: the summary tick mechanism — explicit timer task vs flush-on-batch.
A timer task is the simplest shape; a flush-on-batch shape ties
visibility to traffic volume and may starve under low load.

## Out of scope (explicitly not user stories at v0)

- **Tail-based sampling.** Holds span batches in memory for N
  seconds and applies whole-trace rules. Roadmap C.4 says v1+.
- **PII-scrubbing rules in CUE.** Roadmap C.4 says co-located in
  Sieve and authored in CUE. Defer to v1 (per Q4).
- **Per-tenant or per-service rate.** Needs Aegis (tenant catalogue);
  defer to v1+ (per Q5).
- **Logs severity filtering, metrics aggregation reduction.** Different
  sampling shapes for different signals; defer to v1+ (per Q6).
- **Separate-process binary shape.** v0 is a library; v1+ if the
  tail-sampling window justifies the extra hop (per Q1).
- **Dynamic rate changes at runtime.** v0 reads the env var at
  startup; reload at SIGHUP or runtime API is post-v0.
